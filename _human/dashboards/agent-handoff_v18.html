01Repository architecture

Recommended baseline · 建議的基礎架構

project_name/
├── README.md                    # Human entry point / 人類入口
├── AGENTS.md                    # Agent rules / Agent 操作規則
├── .gitignore                   # Privacy boundary / Git 隱私防線
├── .env.example                # Only when environment variables are needed
│
├── _meta/
│   ├── purpose.md              # Purpose and scope / 目的與範圍
│   ├── roadmap.md              # Priorities / 優先順序
│   ├── handoff.md              # Current status / 最新交接
│   ├── private_blueprint.md    # Local only; never commit
│   ├── public_blueprint.md     # Share-safe reconstruction blueprint
│   └── changelog.md            # Append-only change history
│
├── _agent/
│   └── README.md               # Active prompts, context and workflows index
│
├── skills/
│   └── README.md               # Active skills index
│
├── _human/
│   └── README.md               # Human-facing outputs index
│
├── _pending/
│   └── index.md                # Human review buffer
│
└── <project-specific-folders>/ # app/, src/, public/, tests/, docs/, etc.

Private boundary / 私密邊界
External Agents must not receive .env, Secrets, personal data, private memory, or _meta/private_blueprint.md.

03What belongs in each file?

README.mdPurpose, status, quick start, major functions, and folder guide.
專案用途、狀態、快速開始、主要功能與資料夾導覽。
AGENTS.mdMandatory operating rules, testing requirements, privacy limits, and prohibited actions.
Agent 必須遵守的操作規則、測試要求、隱私限制與禁止事項。
_meta/public_blueprint.mdShare-safe architecture, data flow, environment variable names, and reconstruction steps.
可安全分享的架構、資料流、環境變數名稱及重建步驟。
_meta/handoff.mdLatest verified state, completed work, current issues, and exact next action.
最新確認狀態、完成事項、目前問題與明確下一步。
_agent/workflows/Reusable multi-step procedures only when the project truly needs them.
只有專案確實需要時，才建立可重複使用的多步驟流程。
skills/Project-specific reusable capabilities; store each Skill directly under this folder.
專案專用能力模組；每個 Skill 直接放在此資料夾下。

04Mandatory Agent rules

  • Preserve useful existing files and project-specific structures.
  • Never permanently delete project content.
  • Move uncertain items to _pending/ and update its index.
  • Never read, print, copy, expose, or commit Secrets.
  • Never commit _meta/private_blueprint.md.
  • Never overwrite generated outputs; create the next version.
  • Keep canonical governance filenames unchanged.
  • Do not create optional folders without actual work.
  • Run relevant tests and production builds before completion.
  • Update handoff.md and changelog.md.

Versioned generated outputs

Use Git history for fixed governance files. Use incremental filenames for generated artifacts.

08Q&A

Should I paste the complete architecture into every task?

No. Store the stable rules in the repository. The task prompt should contain only the current objective, acceptance criteria, and special exceptions.
不需要。穩定規則保存在 Repository；任務 Prompt 只放目前目標、驗收標準與特殊例外。

Where should a unique rule for one folder go?

Use a nested AGENTS.md only when that folder genuinely needs different rules. Keep the root file concise.
只有特定資料夾真的需要不同規則時，才使用巢狀 AGENTS.md。

Should private_blueprint.md be sent to Codex?

Only when Codex is operating locally, access is explicitly authorized, and the information is genuinely necessary. Never commit or send it to an external service.
只有 Codex 在本機操作、已明確授權且確實需要時才可讀取；絕不可提交 Git 或傳送給外部服務。
